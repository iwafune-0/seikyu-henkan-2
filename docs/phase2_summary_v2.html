🔧 Git運用ルール（個人開発最適化）

💡 個人開発に最適化

このプロジェクトは個人開発なので、チーム開発で使う複雑なGitルールは不要です。シンプルに運用します。

通常のGit操作（3ステップ）

今後の変更は以下のコマンドで一括処理：

git add . git commit -m "適切なメッセージ" git push -f origin main

各コマンドの意味:

  • git add . - 全ての変更をステージング （記録準備エリアに追加）
  • git commit -m "..." - 変更を記録
  • git push -f origin main - GitHubにアップロード（force push = 強制上書き）

コミットメッセージ規約

変更内容が分かりやすいように、決まった形式で書きます：

type: 簡潔な説明

推奨タイプ:

  • feat: - 新機能追加（例: feat: ログイン画面を追加）
  • fix: - バグ修正（例: fix: ログインエラーを修正）
  • docs: - ドキュメント更新（例: docs: READMEを更新）
  • refactor: - リファクタリング （コードの整理）
  • chore: - その他の変更
  • deps: - 依存関係更新（例: deps: Reactを18.3.0に更新）

機密情報対応プロトコル

⚠️ 重要: パスワードなどをGitHubに公開しない

コミット前に、パスワードやAPIキーなどが含まれていないかチェックします。

コミット前チェックコマンド:

grep -r -i "APIKey\|api_key\|secret\|password\|token\|credential" .

機密情報が見つかった場合:

  1. .gitignore に追加（Gitの管理対象から除外）
  2. git reset でステージングから除外
  3. 環境変数ファイル（.env）へ移行

過去のコミットに含まれていた場合:

  • 直前のコミット: git reset --hard HEAD~1 で削除して再コミット
  • 過去のコミット: git filter-branch で履歴から完全削除

💡 今後の運用方針

日常的なGit操作

変更を加えたら、以下を実行：

git add . git commit -m "変更内容の説明" git push -f origin main

または、Git管理オーケストレーターに「Git操作してください」と依頼すれば自動実行されます。

ブランチ戦略

個人開発のため、シンプルな運用:

  • メインブランチ: main のみ
  • 基本的にブランチ分岐 （作業を分けること） なし
  • 必要に応じて機能ブランチを作成（Phase 10まではほぼ不要）

📖 ブランチとは？

開発作業を「枝分かれ」させて、複数の作業を並行して進められる機能です。ただし、個人開発では基本的に main ブランチ1本だけで十分です。

緊急リセット

複雑になったGit状態をリセットして、最新の状態に戻す方法：

git reset --hard origin/main git clean -fd

⚠️ 注意: ローカルの変更が全て消えるので、慎重に使用してください。
